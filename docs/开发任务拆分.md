# Dify-KB-Eval 开发任务拆分

## 1. 总目标

搭建独立内部知识库评测系统,支持基于 JSONL 金标数据集对 Dify 知识库检索能力做批量评测,把结果持久化为 PostgreSQL 元数据 + 3 个磁盘产物,并支持跨配置的横向对比。

KB-Eval 是独立项目,不接入外部产品菜单,不改外部业务 API 和数据库迁移。

## 2. 分工总览

| 角色 | 负责范围 | 主要产物 |
| --- | --- | --- |
| 前端 | 独立评测页面、历史列表、分析对比、报告展示、评测集编辑器 | `frontend/src/pages/`、`frontend/src/widgets/` |
| 后端 | FastAPI、任务状态、PostgreSQL 元数据、磁盘产物、Dify 透传 | `backend/app.py`、`backend/services/`、`backend/db/` |
| 评测核心 | JSONL 校验、Dify client、指标、报告生成、PDF 解析(MinerU / MarkItDown) | `kb_eval/` |
| 测试 / 评审 | S1720 / 新增厂商型号数据集质量、验收样例、阈值确认 | `datasets/`、`tests/` |

## 3. 当前实现能力清单

按"现在就能用 / 现在还做不到"两栏写,避免分期口吻。

### 3.1 已实现

- 独立 FastAPI 后端 + 同步 SQLAlchemy,三张表 `runs` / `run_summaries` / `run_reports`
- 评测集审核状态机:`unreviewed` / `draft` / `reviewed`,未审核禁止创建 run
- 评测任务后台执行 + 进度实时回写 + 前端每 2s 轮询
- 跑完把 `summary.json` / `report.md` 从磁盘搬到 DB,清理磁盘副本
- 3 个可下载磁盘产物:`results.jsonl` / `results.csv` / `console.log`
- `/api/knowledge-bases` 透传 Dify,前端"选 KB"下拉
- 选完 KB 自动回填 `embedding_model` / `retrieval_model_dict` 到运行表单并只读
- `/api/runs/compare` 按 `(embedding_model, rerank_model, sample_count)` 分组,组内 `best_run_id` 用 Recall@5 → MRR → 耗时打分
- 历史 run 改名(`PATCH /api/runs/{id}`)与对比标签修正(`POST /api/runs/{id}/labels`,独立窄接口)
- 删除运行:软删除 + 一次性 `reports/<id>.deleted-<UTC>/` 备份,运行中拒绝,目录缺失返回 `missing` 保持幂等
- 一键启动脚本自动起 PostgreSQL(`docker compose up -d db`)
- Mock 模式(`VITE_USE_MOCK=true`)脱离后端预览
- 前端六个页面:Home / Runs / RunDetail / RunCompare / Datasets / DatasetEditor

### 3.2 尚未实现(后续候选)

- "评测结论"摘要导出
- 评测集审核状态机的二次打磨(批量操作、冲突提示等)
- LLM-as-judge(尚未排期)

## 4. 后端任务明细

目录:

```text
backend/
  app.py
  schemas.py
  db/
    base.py
    session.py
    models.py
  services/
    run_service.py
    db_store.py
    artifact_store.py
    dataset_generation_service.py
    dataset_edit_service.py
    dataset_review_service.py
    store_protocol.py
```

任务清单:

- `app.py`:创建 FastAPI app、CORS、路由注册、lifespan 加载 `.env` + `init_db`
- `schemas.py`:定义 Pydantic request / response
- `db/session.py`:同步 SQLAlchemy 引擎,`postgresql+asyncpg://` 自动归一为 `postgresql+psycopg://`,未设置 `DATABASE_URL` 时回退到 in-memory SQLite(测试)
- `db/models.py`:`Run` / `RunSummary` / `RunReport` 三张表 + JSON 列 + 软删除列
- `services/db_store.py`:`RunStore` 协议实现,负责创建 / 进度 / 列表 / 对比 / 软删
- `services/artifact_store.py`:3 个磁盘产物的读写、备份、删除
- `services/run_service.py`:创建后台任务、调用 `kb_eval.runner`、写进度、捕获异常、`list_knowledge_bases` 透传
- `services/dataset_generation_service.py`:源文件处理与评测集生成,支持 MinerU / MarkItDown 双后端
- `services/dataset_edit_service.py` + `dataset_review_service.py`:评测集读写、审核、草稿状态机
- 安全处理:过滤 artifact 文件名,禁止路径穿越;`eval_file` 必须在白名单目录内
- Token 处理:`dify_api_key` 不写入报告或下载产物；Dify API 地址和 Key 成对写入后端历史连接配置表，用于下拉复用

优先级:

| 优先级 | 任务 |
| --- | --- |
| P0 | health、datasets、runs create / list / detail |
| P0 | PostgreSQL 元数据 + 3 个磁盘产物 |
| P0 | 调用 Dify 执行真实评测 |
| P1 | artifact 下载、Markdown 报告读取 |
| P1 | 运行中 progress 实时回写 |
| P1 | 评测集审核 / 草稿状态机 |
| P1 | 选 KB 自动回填标签 |
| P2 | 跨配置对比分析页 |
| P2 | 改名 / 改标签独立端点 |
| P2 | 软删除 + 一次性备份目录 |

## 6. 评测核心任务明细

```text

  dify_client.py
  metrics.py
  runner.py
  report.py
  errors.py
```

迁移原则:

- 现有 `scripts/kb_eval.py` 已经实现可用闭环,应拆分复用,不另起一套指标
- CLI、FastAPI 都调用同一套 `runner`
- 指标字段名保持稳定,前端和报告都依赖这些字段

P0 接口:

```python

def run_evaluation(config: EvalConfig, progress: ProgressCallback | None = None) -> EvalRunResult: ...

def write_artifacts(result: EvalRunResult, output_dir: Path) -> None: ...
```

## 7. 报告持久化任务

目录结构:

```text
reports/<run_id>/
  results.jsonl
  results.csv
  console.log
```

`manifest.json` / `summary.json` / `report.md` 在跑完后会被搬到 PostgreSQL 三张表(`runs` / `run_summaries` / `run_reports`),磁盘副本清理。

软删除备份:

```text
reports/<run_id>.deleted-<UTC 时间戳>/
  # 一次性备份,删除原 run 时自动生成
```

失败时仍保留:

```json
{
  "id": "20260609-113000-s1720-top5",
  "status": "failed",
  "error": "Cannot reach Dify at http://localhost/v1/...",
  "created_at": "2026-06-09T11:30:00+08:00",
  "finished_at": "2026-06-09T11:30:03+08:00"
}
```

## 8. 联调顺序

1. 后端先提供 mock API。
2. 前端用 mock API 完成页面闭环。
3. 评测核心拆分完成后,后端替换 mock run_service。
4. 使用 `limit=3` 跑 S1720 小样本。
5. 使用 `limit=0` 跑全量 70 条。
6. 前端核对报告展示和下载。
7. 元数据迁到 PostgreSQL,跑完落 DB,清理磁盘副本。
8. 加横向对比、选 KB 自动回填、软删 + 备份。

## 9. 验收用例

| 用例 | 操作 | 期望 |
| --- | --- | --- |
| 健康检查 | 打开 `/api/health` | 返回 `ok` |
| 数据集列表 | 打开页面 | 能看到 S1720 评测集,显示 review 状态 |
| 小样本评测 | `limit=3`、`top_k=5` | 生成完整产物 + DB 落库 |
| 全量评测 | `limit=0`、`top_k=5` | 70 条主问题评测完成 |
| 同义问法 | 开启 `include_alternatives` | query 数增加 |
| Dify 不可用 | 填错地址 | 运行失败且页面展示错误 |
| 产物下载 | 下载 CSV / JSONL / console.log | Excel 可打开中文 |
| Token 安全 | 提交 token 后检查报告 | 报告不包含 token 明文 |
| 选 KB 自动回填 | 在 KB 下拉里挑一项 | embedding / rerank 标签只读回填 |
| 跨配置对比 | 同一 dataset 多次跑不同 KB | 分析对比页能分组并高亮 best run |
| 软删除 | 删除一次 run | DB 行打 `deleted_at`,磁盘有 `.deleted-<UTC>/` 备份 |
| 评测集未审核 | 用 draft 数据集创建 run | 返回 `DATASET_REVIEW_REQUIRED` |

## 10. 不做事项

- 不注册客户侧菜单
- 不写外部业务数据库
- 不新增客户侧 Alembic migration
- 不在客户侧 API 暴露评测报告
- 不接外部实验追踪平台(不依赖 LangSmith 等)
- 不把 Dify API Key 写入报告、数据库或运行产物
