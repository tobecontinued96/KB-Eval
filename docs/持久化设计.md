# Dify-KB-Eval 报告持久化设计

## 1. 方案选择

当前采用 "PostgreSQL 元数据 + 3 个磁盘产物" 混合持久化,而不是纯文件或纯数据库。

原因:

- 列表 / 详情 / 对比 / 标签修改接口要按 `dataset_id` / `embedding_model` / `rerank_model` / 状态高效过滤,纯文件扫目录跑不动
- 评测明细(`results.jsonl` / `results.csv`)是流式产物,适合直接落盘
- Markdown 报告正文跟元数据一起查比读两个文件省一次 I/O
- 数据库仍属于 `Dify-KB-Eval` 自带,依赖 `docker compose up -d db` 起的 PostgreSQL 16,**不接客户业务数据库**

## 2. 存储分工

| 维度 | 位置 | 说明 |
| --- | --- | --- |
| run 元数据 / 进度 / 核心指标 | PostgreSQL `runs` 表 | 启动时 `Base.metadata.create_all`(`RUN_DB_BOOTSTRAP=true` 时执行);生产设 `RUN_DB_BOOTSTRAP=false` 后由 lifespan 校验 `alembic_version` 必须在 head,否则拒启(`backend/db/session.py:require_alembic_head`) |
| 报告正文 / Markdown | PostgreSQL `run_reports.content` | 跑完从 `report.md` 搬到 DB,磁盘副本删掉 |
| 聚合指标 / 按 scenario 分组 | PostgreSQL `run_summaries.overall` / `by_scenario_type` | 跑完从 `summary.json` 搬到 DB,磁盘副本删掉 |
| 每条查询的检索明细 | `reports/<run_id>/results.jsonl` | `kb_eval.runner` 直写,通过 `GET /api/runs/{id}/artifacts/results.jsonl` 下载 |
| CSV 导出 | `reports/<run_id>/results.csv` | `kb_eval.runner` 直写,UTF-8 with BOM |
| 运行过程日志 | `reports/<run_id>/console.log` | `ArtifactStore.append_log` 直写 |
| 评测集 | `datasets/*.jsonl` + `datasets/generated/*.jsonl` | 草稿落在同名 `.draft.jsonl`,审核元信息落在 `<name>.review.json` |

软删除:删除运行(`DELETE /api/runs/{id}`)是软删除 — DB 行打 `deleted_at` 时间戳 + 把整份 `reports/<id>/` 目录复制到 `reports/<id>.deleted-<UTC 时间戳>/` 作为一次性备份。`running` / `queued` 状态不允许删,目录缺失时返回 200 + `status="missing"` 保持幂等。

## 3. 目录结构

```text
Dify-KB-Eval/
  reports/
    20260609-113000-s1720-top5/
      results.jsonl
      results.csv
      console.log
    20260609-113000-s1720-top5.deleted-20260617T103000/
      # 一次性备份,删除原 run 时自动生成
```

`reports/` 默认不纳入版本管理,只保留 `.gitkeep`。`summary.json` / `report.md` / `manifest.json` 跑完后会被搬到 PostgreSQL,不再作为磁盘产物出现。

## 4. run_id 规则

推荐格式:

```text
YYYYMMDD-HHMMSS-<slug>
```

示例:

```text
20260609-113000-s1720-top5
```

规则:

- 时间使用本机时区
- `slug` 从运行名称生成,只保留字母、数字、短横线、中文字符
- 如果目录已存在,追加 `-2`、`-3`
- ID 写入前会用 SQL `SELECT` 再确认一次没有同名 run,避免两个 slug 撞车

## 5. runs 表(核心元数据)

`backend/db/models.py:Run` 是一切列表 / 详情查询的主索引,任务创建时立即写入一行。

```text
id              String(128)   primary key
name            String(255)
status          String(16)    queued / running / completed / failed / canceled
created_at      DateTime(tz)
started_at      DateTime(tz)?
finished_at     DateTime(tz)?
duration_ms     BigInteger?
dify_base_url   String(512)
dataset_id      String(128)
eval_file       String(512)
top_k           Integer
include_alternatives  Boolean
limit           Integer
sample_ids      JSON          数组
timeout_seconds Integer?
sample_count    Integer       实际跑的样本数
query_count     Integer       含同义问法的 query 总数
progress        JSON          {total_queries, completed_queries, error_queries, current_sample_id}
metrics         JSON          overall 指标镜像,列表接口不走 JOIN
embedding_model String(128)?  对比标签,空串归一为 NULL
rerank_model    String(128)?  对比标签,空串归一为 NULL
error           Text
deleted_at      DateTime(tz)? 软删除
deleted_backup_path  String(1024)?  软删时一次性备份目录
```

索引:`(status, created_at)` / `(created_at)` / `(deleted_at)`。

关联表:

- `run_summaries(run_id, top_k, ks, overall, by_scenario_type)` — 跑完从 `summary.json` 搬过来
- `run_reports(run_id, content, updated_at)` — 跑完从 `report.md` 搬过来

## 6. summary.json 落库形式

`summary.json` 跑完会被解析成 `RunSummary` 写入 PostgreSQL,磁盘副本删除。结构与 `RunSummary` 模型一致:

```json
{
  "top_k": 5,
  "ks": [1, 3, 5],
  "overall": {
    "total_queries": 20,
    "completed_queries": 20,
    "error_queries": 0,
    "empty_result_rate": 0.0,
    "avg_latency_ms": 1230,
    "p95_latency_ms": 2100,
    "document_mrr": 0.78,
    "document_recall@1": 0.55,
    "document_recall@3": 0.82,
    "document_recall@5": 0.91
  },
  "by_scenario_type": {
    "故障恢复": {}
  }
}
```

## 7. results.jsonl(磁盘产物)

`results.jsonl` 每行一条 query 结果,适合二次分析。

关键字段:

| 字段 | 说明 |
| --- | --- |
| `sample_id` | 样本 ID |
| `query_kind` | `primary` 或 `alt-1` |
| `query` | 实际问题 |
| `result_count` | Dify 返回数量 |
| `doc_hit_rank` | 文档命中位次 |
| `section_hit_rank` | 章节命中位次 |
| `keyword_hit_rank` | 关键词命中位次 |
| `latency_ms` | 检索耗时 |
| `error` | 错误信息 |
| `top_results` | Top K 召回详情 |

## 8. results.csv(磁盘产物)

`results.csv` 面向人工分析和 Excel 打开。

编码建议使用 `UTF-8 with BOM`,避免中文乱码。

推荐列:

```text
sample_id,query_kind,scenario_type,topic,query,result_count,doc_hit_rank,section_hit_rank,keyword_hit_rank,latency_ms,error,top1_document,top1_score
```

## 9. report.md 落库形式

`report.md` 跑完会作为整段文本搬到 `run_reports.content`,磁盘副本删除。建议章节:

- 运行配置
- 指标总览
- 按场景指标
- 失败样本 Top 20
- 产物说明

## 10. 写入时序

```mermaid
sequenceDiagram
  participant FE as Frontend
  participant BE as Eval Backend
  participant DB as PostgreSQL
  participant FS as reports/&lt;run_id&gt;/
  participant Runner as Eval Runner
  FE->>BE: POST /api/runs
  BE->>DB: INSERT runs(queued) + INSERT run_reports(content='')
  BE->>FS: mkdir + touch console.log
  BE-->>FE: run_id
  BE->>DB: UPDATE runs SET status='running'
  BE->>Runner: run_evaluation(config)
  Runner-->>FS: 写 results.jsonl / results.csv
  Runner-->>DB: progress 回调 (UPDATE runs.progress)
  Runner-->>FS: 写 summary.json / report.md
  BE->>FS: 读 summary.json / report.md
  BE->>DB: UPDATE runs.metrics + INSERT run_summaries + UPDATE run_reports
  BE->>FS: 删 summary.json / report.md
  BE->>DB: UPDATE runs SET status='completed'
  FE->>BE: GET /api/runs/{run_id}
  BE-->>FE: 详情聚合(全部走 SQL)
```

## 11. 安全要求

- 后端不保存 `dify_api_key`;请求结束后随内存释放
- 不保存任何外部服务的 API Key
- 错误信息需要脱敏 URL query 或 header
- `embedding_model` / `rerank_model` 仅作分组标签使用,不参与检索逻辑
- `eval_file` 路径必须在白名单目录(`Dify-KB-Eval/datasets` 或 `Dify-KB-Eval/../docs`)内,避免任意文件读取
- artifact 下载路径必须落在 `run_dir(run_id)/` 之下,`name` 必须在 `ARTIFACT_FILES` 白名单内,避免路径穿越
