# 知识库评测数据集规范

## 1. 文件格式

评测集使用 JSONL，每行一条样本，编码统一为 `UTF-8`。

推荐命名：

```text
datasets/<vendor>_<model>_<version>.jsonl
```

示例：

```text
datasets/huawei_s1720_v0.1.jsonl
```

机器读取版建议放在：

```text
datasets/<vendor>_<model>_<version>.jsonl
```

人工评审说明或样本审阅记录建议随项目文档或评测集元信息保存，避免引用本机绝对路径。

## 2. 样本字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 样本唯一编号 |
| `vendor` | string | 是 | 厂商，例如 `华为` |
| `model` | string | 是 | 型号，例如 `S1720` |
| `scenario_type` | string | 是 | 场景类型，例如 `故障恢复`、`配置操作` |
| `topic` | string | 是 | 主题，例如 `Console 密码恢复` |
| `difficulty` | string | 否 | 难度，例如 `基础`、`中等`、`高级` |
| `question` | string | 是 | 主问题 |
| `alternative_queries` | string[] | 否 | 同义问法，用于鲁棒性测试 |
| `expected_documents` | string[] | 是 | 期望命中的源文档 |
| `expected_sections` | string[] | 是 | 期望命中的章节标题 |
| `expected_keywords` | string[] | 是 | 期望在召回片段中出现的关键词 |
| `evaluation_focus` | string | 是 | 人工评审关注点 |

## 3. 样本示例

```json
{
  "id": "HW-S1720-EVAL-001",
  "vendor": "华为",
  "model": "S1720",
  "scenario_type": "故障恢复",
  "topic": "Console 密码恢复",
  "difficulty": "基础",
  "question": "华为 S1720 Console 口登录密码忘了，应该怎么恢复？",
  "alternative_queries": [
    "S1720 忘记 console 密码怎么清除？",
    "华为交换机 Console 登录不了，密码丢失后如何处理？"
  ],
  "expected_documents": [
    "01-01 常见系统操作.pdf"
  ],
  "expected_sections": [
    "1.1 Console 口登录密码丢失后如何恢复"
  ],
  "expected_keywords": [
    "Console",
    "BootROM",
    "BootLoad",
    "Clear password for console user"
  ],
  "evaluation_focus": "应命中 Console 口密码恢复章节，并区分远程登录修改密码与 BootROM/BootLoad 清除密码两类路径。"
}
```

## 4. 质量要求

- `id` 全局唯一，建议格式为 `<厂商缩写>-<型号>-EVAL-<三位序号>`。
- `question` 应贴近真实用户问法，不要只复制标题。
- `alternative_queries` 应覆盖口语化、命令型、排障型表达。
- `expected_documents` 使用知识库入库后的源文件名，尽量与 Dify 返回的 `document_name` 一致。
- `expected_sections` 使用源 Markdown 的真实章节标题。
- `expected_keywords` 不应只写厂商/型号，应包含能区分正确章节的命令、协议、配置对象或故障现象。
- `evaluation_focus` 写给人工复核和未来 LLM-as-judge 使用,应说明"命中什么"和"避免误判什么"。

## 5. 覆盖建议

每个型号第一版建议至少覆盖 `50-100` 条样本：

| 类型 | 建议占比 | 示例 |
| --- | --- | --- |
| 故障恢复 | 15%-25% | 密码恢复、配置恢复、异常清理 |
| 配置操作 | 30%-40% | VLAN、接口、链路聚合、ACL |
| 查询诊断 | 20%-30% | 查看状态、查看表项、查看日志 |
| 安全与准入 | 10%-20% | AAA、NAC、IPSG、DHCP Snooping |
| 协议特性 | 10%-20% | STP、OSPF、VRRP、SNMP |

同一主题建议至少包含：

- 1 条直接问法。
- 1 条口语化问法。
- 1 条命令或现象驱动问法。

## 6. 校验规则

后端和 CLI 需要在运行前执行校验：

- 文件存在且为 UTF-8 JSONL。
- 每行是合法 JSON object。
- 必填字段不为空。
- 列表字段必须是 string 数组。
- `id` 不重复。
- `vendor`、`model` 在同一评测集中原则上保持一致；混合评测集必须显式传入 `dataset_id`。
- 空行可以忽略。

## 7. 指标映射

| 数据字段 | 用于指标 |
| --- | --- |
| `expected_documents` | `document_recall@K`、`document_mrr` |
| `expected_sections` | `section_recall@K`、`section_mrr` |
| `expected_keywords` | `keyword_recall@K`、`keyword_mrr` |
| `alternative_queries` | 同义问法鲁棒性对比 |
| `scenario_type` | 按场景聚合指标 |

## 8. 新数据集生产流程

1. 从源文档目录筛选已处理好的 Markdown 文件。
2. 按章节抽取候选问题，优先覆盖高频运维场景。
3. 为每条问题补充源文档、章节、关键词、评估关注点。
4. 人工抽查不少于 `20%` 样本，确认期望文档和章节可被追溯。
5. 使用 `kb_eval` dry-run 校验 JSONL。
6. 首轮 Top5 检索评测后，把明显不可评测或期望错误的样本修正。

## 10. PDF 源文档与 MinerU 处理约定

推荐按知识库名称组织源文档目录：

```text
华为/S1720/
  *.pdf
  md/
    *.md
```

- `source_directory` 使用知识库目录，例如 `华为/S1720`。
- PDF 经 MinerU 处理后的 Markdown 固定保存到同级 `md/` 目录，文件名尽量与 PDF 同名。
- 前端目录上传模式会把文件复制到 `generated_sources/华为/S1720/`，其 Markdown 保存到 `generated_sources/华为/S1720/md/`。
- 型号从目录选择器返回的文件夹名解析，厂商由用户从候选下拉中选择或输入自定义值，最终拼接知识库名称，例如 `华为 S1720`；不需要手工填写源目录。
- 评测集生成器优先复用已有 `md/<同名>.md`，避免每次重复跑 MinerU。
- 生成的 JSONL 放入 `datasets/generated/`，用于现有评测流程直接选择。
- 知识库名称应与 `vendor model` 一致，例如 `华为 S1720`；选择目标知识库时会校验该名称，避免评测跑错知识库。
- MinerU 官方 API 与本地命令共用 PDF 到 Markdown 适配层，JSONL 生成与评测流程保持一致。
