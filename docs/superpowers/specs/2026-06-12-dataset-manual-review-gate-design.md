# 评测集人工审核强制关卡设计

日期：2026-06-12

## 背景

当前系统已经具备评测集草稿与审核能力：生成服务会写入 `<stem>.draft.jsonl`，编辑器可以保存草稿并通过“标记为已审核”提交到主 JSONL，后端会写入 `<stem>.review.json`。但首页发起评测时仍允许选择待审核草稿继续执行，只做弹窗提醒。

用户已确认新规则：评测集生成之后必须经过人工审核，审核通过后才允许发起评测。

## 目标

- 自动生成的评测集默认进入 `draft` 状态。
- `draft` 和 `unreviewed` 状态的数据集不能发起评测。
- 用户必须在评测集编辑器中完成检查，并点击“标记为已审核”。
- 只有 `reviewed` 状态的数据集可以创建评测运行。
- 前端给出清晰引导，后端提供最终约束，防止绕过 UI 直接调用 API。

## 非目标

- 不引入多人审批、审批流转、角色权限或电子签名。
- 不改变 JSONL 样本格式。
- 不改变现有草稿文件与审核元信息文件命名规则。
- 不自动判定样本质量是否合格，人工审核仍由审核人负责。

## 设计方案

采用前后端双重强制关卡。

后端在 `POST /api/runs` 创建评测运行前读取 `eval_file` 的审核状态。若状态不是 `reviewed`，直接拒绝创建运行，返回业务错误，例如 `DATASET_REVIEW_REQUIRED`，并在错误详情中包含 `eval_file`、`review_status`、`draft_path`。这保证即使绕过前端，也不能使用未审核数据集跑评测。

前端在首页评测表单中根据当前选中数据集的 `review_status` 控制交互。状态为 `draft` 或 `unreviewed` 时，“开始评测”按钮禁用，页面展示待审核提示，并提供“去审核”入口跳转到 `/datasets/<path>/editor`。状态为 `reviewed` 时保持现有提交流程。

生成页面在生成成功后强调“已生成草稿，待人工审核”，主行动按钮应指向编辑器审核，而不是引导用户去首页发起评测。数据集概览继续展示审核状态、草稿路径、审核人和审核时间。

## 状态规则

`draft`：存在 `<stem>.draft.jsonl`，编辑器优先展示草稿内容；保存动作继续写回草稿；只有“标记为已审核”会写入主 JSONL。

`reviewed`：不存在草稿，存在 `<stem>.review.json` 且状态为 `reviewed`；允许创建评测运行。

`unreviewed`：旧版样本或缺少审核元信息；不允许创建评测运行。用户可以打开编辑器，确认当前主 JSONL 内容后提交审核，系统写入审核元信息。

## 数据流

1. 用户上传 PDF 或 Markdown 并生成评测集。
2. 生成服务写入草稿文件和 `draft` 审核元信息。
3. 前端刷新数据集列表，显示“草稿待审核”。
4. 用户进入编辑器检查和修改样本。
5. 用户点击“标记为已审核”。
6. 后端校验 rows，通过后写入主 JSONL、删除草稿、写入 `reviewed` 元信息。
7. 首页选择该数据集后允许发起评测。
8. 后端创建运行前再次检查状态，只允许 `reviewed`。

## 错误处理

后端拒绝未审核数据集时返回稳定错误码 `DATASET_REVIEW_REQUIRED`。错误消息面向用户说明“评测集尚未通过人工审核，请先在评测集编辑器中标记为已审核”。详情字段包含：

- `eval_file`
- `review_status`
- `draft_path`

前端若收到该错误，应展示同样含义的提示，不创建运行，不进入历史详情页。

## 测试与验证

- 后端单元或集成测试覆盖 `draft`、`unreviewed`、`reviewed` 三种状态创建运行的结果。
- 前端构建通过，首页对未审核数据集禁用提交按钮，并提供审核入口。
- 手工验证流程：生成评测集后不能直接发起评测；进入编辑器提交审核后可以发起评测。
- 继续确认已有编辑器保存草稿、提交审核、列表状态展示不回退。

## 迁移影响

已有未审核 JSONL 会被视为 `unreviewed`，不能直接跑评测。迁移路径是打开编辑器并提交审核，不需要修改样本内容即可写入审核元信息。

## 风险

- 旧数据集第一次使用前多一步审核操作，短期会改变已有习惯。
- 如果前端列表没有及时刷新，用户可能看到旧状态；后端强制校验可以避免真正创建未审核运行。
- 若审核元信息文件损坏，系统会按现有读取逻辑降级处理，需要在实现中确认不会误放行。
