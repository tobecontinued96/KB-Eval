## 工作记录：AINE2-KB-Eval 支持 Dify 直连

### 需求背景
- 当前 AINE2-KB-Eval 通过 AINE2-Gateway 调用 Dify 背后的知识库检索能力，用户希望评测工具可以绕过 Gateway，直接连接 Dify Knowledge Base API。
- 直连后需要保持现有评测指标、报告、前端选 KB 流程尽量不变，同时避免把 Dify API Key 写入数据库、localStorage 或报告产物。

### 问题与分析
- 现有调用链集中在 `kb_eval/gateway_client.py` / `kb_eval/async_gateway_client.py`，协议固定为 Gateway 的 `/api/v1/knowledge-bases` 与 `/knowledge-bases/{id}/retrieve`。
- Dify 官方 Knowledge Base API 使用 `GET /datasets`、`POST /datasets/{dataset_id}/retrieve`，鉴权为 `Authorization: Bearer <API Key>`，返回结构为 `records[].segment`，与当前评测指标消费的 `content`、`document_id`、`document_name`、`score` 不一致。
- 为避免影响已有 Gateway 评测，采用“双模式”方案：新增 `connection_type=gateway|dify`，默认仍为 Gateway；Dify 模式通过适配层把返回结构归一化成现有 chunk 格式。
- 因为 Dify API Key 必填且不落库，Dify 直连运行在后端内存中的 background task 执行，避免 runner 子进程从 DB 恢复配置时拿不到密钥或需要持久化密钥。

### 实现方案
- 新增同步和异步 Dify client，支持 Dify 知识库列表、dataset 自动匹配、名称校验和检索结果归一化。
- 在 `EvalRunConfig` 增加 `connection_type`，同步 runner 与异步 runner 根据连接方式选择 Gateway client 或 Dify client。
- 后端 `/api/knowledge-bases` 增加 `connection_type` 查询参数，创建运行请求体增加 `connection_type` 字段。
- 前端评测配置表单新增“连接方式”选择，支持 `AINE2 Gateway` 与 `Dify 直连`；Dify 模式下地址默认提示为 `http://127.0.0.1:5001`，Token 输入显示为 `Dify API Key`。
- 报告和文档文案从固定 Gateway 改为更通用的连接方式 / API 地址描述。

### Backend / Core 改动
- `kb_eval/dify_client.py`
  - 新增 Dify 同步适配器，处理 `/v1/datasets` 和 `/v1/datasets/{dataset_id}/retrieve`。
  - 将 Dify `records[].segment` 映射为现有评测指标需要的 chunk 结构。

## 工作记录：Dify API 地址 + Key 自动保存

### 需求背景

- 评测配置页的 `Dify API 地址` 和 `Dify API Key` 两个字段当前都不落盘，每次刷新都要重新输入。
- Key 输入框 placeholder 仍写着"不落 localStorage"，与"不持久化"的当前行为一致；用户希望刷新后能自动回填上次填写的一对值，后续仍可手动修改其中一项。

### 问题与分析

- 现有 `RUN_PREFERENCES_STORAGE_KEY`（`dify-kb-eval:home:run-preferences:v1`）只持久化 `top_k / limit / timeout_seconds`，由 `applyRunPreferences` 合并，且 `writeRunPreferences` 仅在 `createRun` 成功后调用。
- 凭据对不应复用 `applyRunPreferences`：其函数职责被有意收窄为运行参数，扩展它会把 token 拉到运行偏好合并路径上，污染安全模型。
- "URL 与 Key 一一对应"需要"原子对"语义：blur 时无论只编辑了哪个字段，都把当前 URL + 当前 Key 一起落盘，避免两者错位。
- 已经有 `useEffect` 监听 `form.gateway_base_url` 和 `form.gateway_token` 变化并触发 `listKnowledgeBases` 300ms 防抖请求，所以"挂载即回填"会自然触发 KB 列表重拉，无需改 effect。
- token 以明文写入 localStorage 是用户明确接受的取舍，提交信息中说明是有意为之。

### 实现方案

- 新增独立 helper 模块 `frontend/src/pages/homeDifyCredentials.ts`，对外暴露 `DIFY_CREDENTIALS_STORAGE_KEY` (`dify-kb-eval:home:dify-credentials:v1`)、`readDifyCredentials`、`writeDifyCredentials`、`applyDifyCredentials`、`isDifyCredentialsUnchanged`。
- 风格沿用 `readRunPreferences / writeRunPreferences / applyRunPreferences`：`typeof window === "undefined"` SSR 守卫 + try/catch 静默吞错 + JSON.parse，并增加 schema 校验（缺 `url`/`key` 字段、非对象、损坏 JSON 一律返回 `null`），防止手改 localStorage 把表单搞崩。
- Home 表单初始化处套一层 `applyDifyCredentials(defaultForm, readDifyCredentials())`，顺序固定为"先凭据后偏好"，避免未来贡献者翻转顺序导致运行偏好被默认 URL/Key 覆盖。
- 在 `Dify API 地址` 与 `Dify API Key` 两个输入上各加 `onBlur={handleDifyCredentialsBlur}`，共享同一 handler。
- `handleDifyCredentialsBlur` 逻辑：
  - 取 `form.gateway_base_url` 与 `form.gateway_token` 当前值；
  - 两字段 trim 后全空 → 跳过（防止清空刷新后丢失历史凭据）；
  - 与已保存值逐字段比较完全一致 → 跳过（避免重复 blur 触发无意义 IO）；
  - 否则把当前一对值整体写回 localStorage。
- placeholder 文案由 `Knowledge Base API Key，临时输入，不落 localStorage` 改为 `Dify API Key（已记住，可手动改）`，与新行为一致。

### Frontend 改动

- `frontend/src/pages/homeDifyCredentials.ts`
  - 新增。导出 `DIFY_CREDENTIALS_STORAGE_KEY`、`DifyCredentials` 接口以及四个 helper。

  - `useState` 初始化器套一层 `applyDifyCredentials`，顺序在注释里写死。

  - placeholder 文案替换。
- `frontend/src/pages/homeDifyCredentials.test.mjs`
  - 新增 8 个测试：round-trip、空存储、损坏 JSON、缺字段、非对象、SSR 守卫、`applyDifyCredentials` 合并保留其他字段、`isDifyCredentialsUnchanged` 匹配/不匹配/`null` 场景。
- `frontend/package.json`
  - `test:helpers` 脚本中追加 `homeDifyCredentials.ts` 编译与 `.test.mjs` 运行。

### 验证

  - 结果：通过。仍输出既有 dynamic import 提示，不影响构建。

  - 结果：30/30 通过，其中 `homeDifyCredentials.test.mjs` 8 个新用例全部通过。
